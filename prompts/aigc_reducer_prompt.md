# 学术表达质量优化提示词指南(增强版)

> 本文档整合 AI 写作模式识别方法，用于定位论文中的模板化表达、机械改写痕迹和语义漂移问题，并辅助进行学术表达质量优化。
>
> **核心声明**：改写目标是保持语义准确、术语稳定和论文表达严谨，而不是追求工具分数或制造不真实的写作特征。

---

## 一、学术表达质量复查维度

### 1.1 复查维度

论文审校时重点关注以下表达质量问题：

| 检测维度 | AI 文本特征 | 人类文本特征 |
|---------|------------|-------------|
| **困惑度(Perplexity)** | 低(语言模型可预测) | 高(表达多样不可预测) |
| **突发性(Burstiness)** | 低(句长均匀) | 高(句长波动大) |
| **词汇多样性(TTR)** | 低(词汇重复率高) | 高(词汇丰富) |
| **过渡词密度** | 高(模板化过渡) | 低(自然过渡) |
| **句式模式** | 整齐(偏好总分总) | 多样(多种结构交替) |

## 二、软件工程方向毕业论文风格要求

- 在保证语义不变的前提下进行重写，而不是简单同义替换
- 避免大量使用“系统应……从而……”这类标准答案式句式
- 控制句式多样性，使文本符合软件工程方向毕业论文的客观表述习惯
- 适当保留自然表达，避免过度压缩造成表达生硬
- 允许使用自然承接语和少量轻冗余词，让段落转场更接近人工写作节奏
- 信息密度过高的句子应先拆出主干，再补一层解释说明，避免一句话塞入过多职责、动作和目标
- 删除口语化、评价性或宣传性表达，改为客观陈述
- 不引入额外技术术语或扩展内容
- 保持论文表达严谨，不使用聊天式口语或夸张化表述

## 三、核心规则速查(5 条铁律)

> **执行任何改写前，牢记这 5 条核心原则：**

| # | 规则 | 说明 |
|---|------|------|
| 1 | **区分过渡词类型** | 压缩模板化转接词，但允许自然承接语帮助段落转场 |
| 2 | **控制轻冗余** | 删除机械废话，保留少量“通常、往往、也会”等缓冲词 |
| 3 | **拆解高密度句** | 对职责、流程、目标塞在一句里的句子，先拆主干再补解释层 |
| 4 | **删除冗余「的」** | 机械堆叠的「的」可删则删，过长定语保留以维持可读性 |
| 5 | **打破公式结构** | 避免二元对比、戏剧性分段、修辞性设置 |
| 6 | **变化节奏** | 混合句子长度，两项优于三项，段落结尾多样化 |
| 7 | **信任读者** | 不做口号式强调，但必要时补一层解释帮助理解 |
| 8 | **删除金句** | 如果听起来像可引用的语句，重写它 |

## 三、章节类型策略矩阵(执行前先判定)

> **执行顺序**：先识别章节类型，再决定改写强度；摘要、总结与展望、致谢默认按保守策略处理。

| 章节类型 | 本章目标 | 优先动作 | 禁用动作 |
|---------|---------|---------|---------|
| 摘要 | 保留三段逻辑、目标/方法/结果稳定 | 删除模板词、轻度句长波动、精简冗余「的」字 | 设问句、强主观表达、明显微瑕疵、结构大改 |
| 绪论/文献综述 | 降低模板味并保留综述体 | 变化引用句式、重点文献详写、删套话 | 强口语化、过多成语、脱离引用的主观发挥 |
| 技术基础/关键技术 | 保留术语准确性 | 写清机制、减少空泛解释、保护术语一致性 | 术语漂移、夸张修辞、无依据扩写 |
| 系统设计 | 突出模块/表结构/架构关系 | 注入模块名、表名、字段、层次信息 | 空泛优势总结、脱离事实发挥、口语式设问 |
| 系统实现 | 突出实现路径和接口调用链 | 补接口路径、关键步骤、流程细节 | 无依据性能评价、整章排比、口语点评 |
| 系统测试/实验分析 | 保留测试条件、数据和结论解释 | 先给结果再解释、保留指标、补边界条件 | 删除数字证据、改成空泛评价、统一句长 |
| 总结与展望 | 保持克制，避免过度修饰 | 删套话、减排比、保留局限与后续工作 | 过强改写、主观感叹、强行制造表达瑕疵 |
| 致谢 | 真诚自然，避免感谢模板化 | 压缩客套空话、保持对象顺序 | 技术化表达、设问句、机械排比 |

### 3.1 改写范围控制(强制)

1. 优先依据 `scripts/aigc/detect.py` 的 `high_risk_paragraphs` 结果定位模板化、句式均匀或表达生硬段落。
2. 先改模板化过渡词、均匀句长、明显总分总结构，再考虑更深层改写。
3. 若局部修正已改善表达自然度，不得继续扩写或整章重写。
4. 摘要、总结与展望、致谢即使模板化特征较明显，也要优先保留学术性与自然性。

## 四、特征消除策略(分级执行)

### P0 优先级(必做)

#### 3.1.1 消除模板化过渡词

**禁用词汇及替换方案：**

| 禁用词汇 | 替换方案 |
|---------|---------|
| 首先…其次…最后 | 直接陈述要点，或用「一方面…另一方面」 |
| 此外 | 另外、同时、与之相伴 |
| 综上所述 | 总体而言、概括来说、总的来看 |
| 值得注意的是 | 需要关注的是、值得关注的是 |
| 不可否认 | 必须承认、诚然 |
| 由此可见 | 这意味着、这说明、不难看出 |

**示例：**

❌ **原文(AI 特征明显)**：
> 首先，大数据技术为精准营销提供了数据基础。其次，大数据技术实现了消费者画像的精准构建。最后，大数据技术支撑了营销决策的优化。

✅ **改写后**：
> 大数据技术对精准营销的支撑体现在三个层面：数据基础的奠定、消费者画像的构建、营销决策的优化。

#### 3.1.2 删除冗余「的」字 ⭐ 新增

> **核心理念**：AI 生成的文本往往「的」字泛滥，人类写作更倾向于精简。删除不影响句意的「的」，可以让句子更简洁有力。

**常见删减模式：**

| 原文 | 改写 | 说明 |
|-----|------|------|
| 重要**的**意义 | 重要意义 | 形容词+名词，可删 |
| 精准**的**构建 | 精准构建 | 动词+名词，可删 |
| 有效**的**方案 | 有效方案 | 形容词+名词，可删 |
| 显著**的**提升 | 显著提升 | 形容词+动词，可删 |
| 广泛**的**应用 | 广泛应用 | 形容词+动词，可删 |
| 企业**的**发展 | 企业发展 | 名词+动词，可删 |

**完整示例：**

❌ **原文(「的」字泛滥)**：
> 大数据技术**的**快速发展为企业**的**精准营销提供了重要**的**技术支撑。通过对用户行为数据**的**采集与分析，企业可以实现消费者画像**的**精准构建。

✅ **改写后**：
> 大数据技术快速发展为企业精准营销提供了重要技术支撑。通过对用户行为数据采集与分析，企业可实现消费者画像精准构建。

#### 3.1.3 增加句长波动

**目标**：句长标准差 > 10

| 操作 | 示例 |
|------|-----|
| 短句强调 | 数据是核心。(6字) |
| 长句阐述 | 在市场竞争日趋激烈、消费者需求日益多元化的背景下，精准营销逐渐成为企业获取竞争优势的关键策略。(45字) |
| 穿插使用 | 市场在变。企业必须适应变化才能生存。而在适应过程中，数据驱动决策的理念逐渐深入人心，成为数字化转型的重要支撑。 |

#### 3.1.4 长短句交织规则 ⭐ 新增

> **核心理念**：句长波动不是把句子机械拉长或切短，而是让不同长度的句子承担不同论述功能。短句用于点题、转折或强调，长句用于解释系统流程、实验条件或设计原因，中句用于承接和收束。

**执行规则：**

1. 每个问题段落至少包含两类句长：短句（15 字以内）、中句（16-30 字）或长句（30 字以上）。
2. 避免连续 3 句长度接近；如果三句都集中在 18-25 字之间，应拆出短句或合并成长句。
3. 推荐节奏为“短句点题 → 长句解释原因或过程 → 中句收束”。
4. 摘要、总结与展望只做轻度波动；系统设计、系统实现、系统测试可更明显地使用长短句交织。

#### 3.1.5 段落结构多样化

| 结构类型 | 示例开头 |
|---------|---------|
| 问题引入式 | 「为什么精准营销越来越重要？」 |
| 数据引入式 | 「2023 年市场规模达到 500 亿元。」 |
| 案例引入式 | 「以某电商平台为例。」 |
| 引用引入式 | 「张三(2023)指出：『…』」 |
| 对比引入式 | 「传统营销与精准营销的区别在于…」 |

### P1 优先级(建议)

#### 3.2.0 祛除八股文特征 ⭐ 新增

AI 偏好的“八股文”通常表现为：背景意义开头、问题挑战过渡、解决方案展开、价值总结收尾。改写时不得只替换连接词，而应改写段落组织方式。

| 八股式写法 | 改写方向 |
|---|---|
| “本文旨在……” | 直接说明研究对象或系统问题 |
| “具有重要意义” | 写清具体影响或使用价值 |
| “提升用户体验” | 写成用户操作路径、系统响应和结果呈现 |
| “首先、其次、最后” | 改为自然叙述，重点内容详写，次要内容带过 |
| “综上所述” | 用具体结论收束，不做口号式总结 |

**处理顺序：** 先判断段落是否存在固定套式；若存在，先重组段落，再处理词语和句式。

#### 3.2.1 增加主观性表达

| 类型 | 表达示例 |
|------|---------|
| 第一人称 | 笔者、本文作者、研究者 |
| 观点表达 | 笔者认为、据笔者观察、在笔者看来 |
| 推测语气 | 据推测、可能意味着、或许可以认为 |
| 质疑语气 | 是否意味着…？值得进一步探讨 |

#### 3.2.2 增加逻辑不完美感

| 操作 | 示例 |
|------|-----|
| 添加转折 | 「虽然效果显著，但也存在一定局限。」 |
| 添加让步 | 「当然，这一结论还需更多实证支持。」 |
| 添加质疑 | 「这一现象背后的原因是什么？值得深入思考。」 |
| 承认局限 | 「限于数据可得性，本研究未能覆盖更多样本。」 |

### P2 优先级(表达自然度控制)

> **核心理念**：论文表达过于工整、均匀和精确时，容易暴露出机械改写痕迹。表达自然度控制的目标，是在保持语义和术语准确的前提下，让句式、论述密度和引用方式更符合毕业论文写作习惯。
>
> **控制建议**：每千字只做少量自然化调整，过少难以改善机械感，过多会显得刻意。

#### 3.3.1 近义词漂移与选词波动

> 在学术语域内选择 **不太常见但完全正确** 的近义词，同时混入适量的模糊修饰词，模拟人类「不总是选最精确用词」的特征。

**近义词漂移对照表：**

| 常规表达 | 可选表达(学术可接受) |
|-----------------|-------------------|
| 应用场景 | 实施范畴、应用语境、实践领域 |
| 研究方法 | 研究路径、分析框架、探究手段 |
| 性能提升 | 性能增益、效能改善、指标优化 |
| 数据显示 | 统计反映、实验结果揭示、数据表明 |
| 解决方案 | 应对策略、处置路径、化解思路 |
| 存在问题 | 面临困境、暴露短板、有待改进 |

**选词波动技巧：**

| 技巧 | 机械写法 | 自然写法 |
|------|--------|---------|
| 混入模糊修饰 | 「显著提升了用户留存率」 | 「用户留存率有了比较大的改善」 |
| 弱化断言 | 「该方案有效解决了…」 | 「该方案在一定程度上缓解了…」 |
| 强弱交替 | 每句都用「显著」「关键」 | 「明显」「较为突出」「有一定…」交替 |

**使用规则：**
1. 同一概念在全文中使用 2-3 种说法，前后自然切换
2. **专业术语不得漂移**(如 Spring Boot、卷积神经网络等必须统一)
3. 标题和摘要仍用规范表达，漂移主要发生在正文段落中
4. 数据描述和结论部分仍需保持用词精确

#### 3.3.2 论述密度不均

> 人类写论文一定会有「我更熟悉这个部分」的痕迹。刻意让论述篇幅不对称，模拟人类注意力分布的不均匀。

| 场景 | 过度对称写法 | 更自然的写法 |
|------|------------------|---------------------|
| 三个论点展开 | 每个各 200 字 | 第一个 300 字、第二个 150 字、第三个 80 字 |
| 文献综述 | 每篇文献评述 2-3 句 | 重点文献详述 5-6 句，其余 1 句带过 |
| 对比分析 | 双方各 150 字 | A 方案 250 字，B 方案 100 字 |

#### 3.3.3 语法层微瑕疵

> 不是语法错误，而是引入人类论文中常见的 **语法偏好与写作习惯**。

| 类型 | 说明 | 示例 |
|------|------|------|
| 长定语不拆分 | 学生论文中高频出现 | 「基于 Spring Boot 框架的面向在线教育场景的智能知识库系统」 |
| 虚词微冗余 | 偶尔多一个「也」「就」 | 「这也在一定程度上说明了…」 |
| 逗号长句 | 用逗号代替句号，形成复句 | 「系统采用 B/S 架构，前端使用 Vue，后端基于 Spring Boot，数据库选用 MySQL。」 |
| 语态不统一 | 主动被动偶尔切换 | 前句「本文采用…」，后句「数据被分为…」 |

**安全边界：** 每段最多 1-2 处；摘要和结论保持规范；不影响语义理解。

#### 3.3.4 引用风格波动

> 同一篇论文中引用方式微妙不一致，模拟人类分批写作的真实痕迹。

| 维度 | 统一写法 | 波动写法 |
|------|------------------|---------------------|
| 引用句式 | 全用「张三(2023)指出…」 | 交替使用「张三(2023)指出…」和「已有研究表明…(张三，2023)」 |
| 引用深度 | 每篇文献都评述 2 句 | 重点文献详述 4-5 句；其余只在括号内带过 |
| 个人评价 | 从不评价引用文献 | 偶尔加入「…虽有参考价值，但样本量偏小」 |
| 引用密度 | 每段均匀 2-3 处引用 | 熟悉的部分引用密集，不熟悉的部分纯靠论述 |

#### 3.3.5 表达自然度学科适配

| 学科类型 | 近义词漂移 | 论述不对称 | 语法微瑕疵 | 引用波动 | 总密度/千字 |
|---------|----------|----------|----------|--------|-----------| 
| 计算机 / 理工科 | 低(术语多) | 中 | 中 | 高 | 3 个 |
| 管理 / 经济学 | 中 | 高 | 中 | 高 | 4-5 个 |
| 文学 / 教育学 | 高 | 高 | 低 | 中 | 4-5 个 |
| 法学 / 政治学 | 中 | 中 | 低 | 高 | 3-4 个 |
| 医学 / 生物学 | 低(术语多) | 中 | 低 | 高 | 3 个 |

#### 3.3.6 自然过渡词与解释性承接 ⭐ 新增

> **核心理念**：降低 AIGC 痕迹并不等于完全删掉过渡词。真正需要压缩的是“因此、此外、综上所述”这类可预测模板连接；在功能说明、系统设计和测试分析中，适度加入自然承接语，反而能让段落更像人工写作。

| 使用场景 | 推荐承接语 | 作用 |
|---|---|---|
| 从职责转到具体动作 | 具体来说、放到实际使用里看 | 把抽象职责落到操作层面 |
| 从功能转到运行过程 | 对应到系统运行过程、在平台持续运行时 | 让说明贴近系统场景 |
| 从结论转到解释 | 换句话说、也就是说、从这个角度看 | 给高密度句补解释层 |
| 从主功能转到补充功能 | 另外、相对来说、另一方面 | 补充次要信息，避免机械列点 |
| 从问题转到处理方式 | 遇到这种情况时、通常需要 | 引出处理逻辑 |

**使用规则：**
1. 每段可加入 1-2 个自然承接语，但不能句句都加。
2. 优先使用“具体来说、换句话说、放到实际使用里看”等解释型承接，少用“首先、其次、最后”。
3. 承接语后面必须有真实内容，例如权限边界、操作步骤、系统响应或结果呈现；不能只接空泛总结。
4. 摘要中少用，需求分析、系统设计、系统实现和系统测试中可适度增强。

#### 3.3.7 可控轻冗余 ⭐ 新增

> **核心理念**：本科论文不需要每句话都压缩到最短。少量“通常、往往、也会、在一定程度上”等缓冲词，可以打散过于规整的机器节奏；但这类词必须服务于承接和解释，不能变成废话堆砌。

| 可保留的轻冗余 | 适用位置 | 示例 |
|---|---|---|
| 通常、往往 | 描述一般运行情况 | 管理员通常会先查看用户状态，再调整权限配置。 |
| 也会、还会 | 补充同类动作 | 系统会记录配置变化，也会保留对应操作日志。 |
| 相对来说、从这个角度看 | 解释判断依据 | 从这个角度看，权限管理并不只是账号增删。 |
| 在一定程度上、比较 | 弱化过满判断 | 该设计在一定程度上降低了误操作风险。 |
| 具体来说、简单理解 | 引出解释层 | 具体来说，这部分工作包括账号维护、角色分配和权限回收。 |

**边界：**
- 保留轻冗余，不等于保留“具有重要意义、发挥重要作用、提升用户体验”等模板空话。
- 技术术语、接口路径、测试数据附近仍需精确，不为了自然化而增加模糊词。
- 同一段轻冗余词不超过 2-3 处，避免显得啰嗦。

#### 3.3.8 高信息密度句拆句解释 ⭐ 新增

> **核心理念**：AIGC 高风险句常常把角色、职责、目标和价值压进一个长句里。处理时不要只替换词语，应先拆出主干，再补一句解释层，让读者能看到作者对系统场景的理解。

**适用对象：**
- “作为……核心角色，其主要职责在于……”这类角色职责句
- 一句话同时包含配置、调度、安全、监控等多个动作的功能句
- 条款式功能说明中每条都高度对称、信息过满的句子
- 测试结论中同时塞入指标、原因和意义的句子

**处理步骤：**
1. 抽主干：先写清“谁负责什么”或“系统完成什么”。
2. 拆动作：把多个职责拆成 2-3 个具体动作。
3. 补解释：用“具体来说、放到实际运行中看、换句话说”等承接语解释这些动作为什么需要放在一起。
4. 收边界：最后保留术语、编号和原有功能范围，不新增原文没有的接口、表或指标。

**示例：**

❌ **高信息密度原句：**
> 系统管理员作为系统运行与安全保障的核心角色，其主要职责在于对平台进行全局配置与资源调度，确保系统的稳定性、安全性与可控性。

✅ **拆句解释后：**
> 系统管理员是平台稳定运行的重要保障。具体来说，该角色需要处理系统参数配置、资源分配和日常调度等工作。放到实际运行中看，这些操作并不是孤立存在的，权限变化、资源占用和安全控制往往会相互影响，因此需要由管理员统一维护。

**改写要点：**
1. “核心角色”先拆成“重要保障”，降低套话感。
2. “全局配置与资源调度”拆成参数配置、资源分配、日常调度。
3. 增加“具体来说”“放到实际运行中看”等自然承接语。
4. 保留职责边界，不编造额外系统功能。

#### 3.3.9 ⛔ 红线清单

| 手段 | 后果 |
|------|------|
| 口语标记(对吧？是吧？你看) | 破坏学术语体，导师直接打回 |
| 故意写错别字或病句 | 降低论文质量，影响成绩 |
| 网络用语(yyds、内卷、破防) | 不符合学术规范 |
| 核心论点逻辑故意断裂 | 影响论文整体评价 |
| 引用虚假或不存在的文献 | 学术不端，可能导致论文不通过 |
| 篡改数据或实验结果 | 严重学术不端 |

## 四、成语替换策略 ⭐ 创新功能

> **核心理念**：成语是汉语言文化的精华，具有独特的人类思维印记。AI 模型倾向于使用通用的同义词替换，而人类更喜欢用成语表达丰富的含义。利用成语替换同义词，可以有效消除「AI 味道」。

### 4.1 成语替换原理

| 特征 | AI 表达 | 成语替换后 |
|------|---------|-----------|
| 可预测性 | 高(统计概率高) | 低(成语选择多样化) |
| 信息密度 | 低(需多个词表达) | 高(四字浓缩) |
| 人类特征 | 低(模板化) | 高(文化印记) |
| 句长波动 | 小 | 大(成语打破节奏) |

### 4.2 成语替换对照表

#### 表示「重要」的成语替换

| AI 高频表达 | 可替换成语 |
|------------|-----------|
| 非常重要 | 举足轻重、至关重要、不可忽视 |
| 具有重要意义 | 意义深远、影响深远、事关重大 |
| 关键作用 | 中流砥柱、举足轻重、关键所在 |
| 核心地位 | 重中之重、核心所在、关键一环 |

#### 表示「困难/问题」的成语替换

| AI 高频表达 | 可替换成语 |
|------------|-----------|
| 面临困难 | 步履维艰、困难重重、荆棘丛生 |
| 存在问题 | 美中不足、白璧微瑕、问题凸显 |
| 遇到挑战 | 迎难而上、面临考验、挑战严峻 |
| 难以解决 | 积重难返、困难重重、阻力重重 |

#### 表示「发展/变化」的成语替换

| AI 高频表达 | 可替换成语 |
|------------|-----------|
| 快速发展 | 日新月异、蒸蒸日上、突飞猛进 |
| 持续增长 | 蒸蒸日上、稳步增长、节节攀升 |
| 变化很大 | 翻天覆地、今非昔比、沧海桑田 |
| 不断创新 | 推陈出新、革故鼎新、与时俱进 |

#### 表示「效果/结果」的成语替换

| AI 高频表达 | 可替换成语 |
|------------|-----------|
| 效果显著 | 立竿见影、效果斐然、成效卓著 |
| 取得成果 | 收获颇丰、硕果累累、初见成效 |
| 得到验证 | 不证自明、事实胜于雄辩 |
| 达成目标 | 如愿以偿、圆满完成、达成所愿 |

#### 表示「关系/联系」的成语替换

| AI 高频表达 | 可替换成语 |
|------------|-----------|
| 密切相关 | 息息相关、休戚与共、唇齿相依 |
| 相互影响 | 相辅相成、互为因果、息息相关 |
| 紧密联系 | 千丝万缕、密不可分、盘根错节 |
| 协调发展 | 齐头并进、协同推进、并行不悖 |

#### 表示「分析/研究」的成语替换

| AI 高频表达 | 可替换成语 |
|------------|-----------|
| 深入分析 | 抽丝剥茧、追根溯源、深入浅出 |
| 仔细研究 | 潜心研究、深入探讨、追本溯源 |
| 全面了解 | 了如指掌、如数家珍、知根知底 |
| 总结归纳 | 提纲挈领、纲举目张、总揽全局 |

### 4.3 成语替换原则

| 原则 | 说明 |
|------|------|
| **语体适配** | 选择书面语成语，避免口语化成语 |
| **语境匹配** | 成语含义需与原文语境相符 |
| **专业保护** | 专业术语附近不使用成语替换 |
| **适度使用** | 每段不超过 2-3 个成语，避免堆砌 |

### 4.4 成语替换示例

❌ **原文(AI 特征明显)**：
> 大数据技术在精准营销中的应用具有非常重要的意义。首先，它能够快速地处理海量数据。其次，它能够有效地提高营销效率。最后，它能够显著地改善用户体验。

✅ **改写后(成语替换)**：
> 大数据技术之于精准营销，可谓**举足轻重**。海量数据的处理**立竿见影**，营销效率的提升**水到渠成**，用户体验的改善**成效卓著**。三者相辅相成，共同构建了精准营销的技术基石。

**改写要点**：
1. ✅ 「非常重要」→「举足轻重」(成语替换)
2. ✅ 「快速地处理」→「立竿见影」(成语替换)
3. ✅ 「有效地提高」→「水到渠成」(成语替换)
4. ✅ 「显著地改善」→「成效卓著」(成语替换)
5. ✅ 消除「首先…其次…最后」模板
6. ✅ 添加「相辅相成」成语收尾

## 五、个性与灵魂注入

> **避免 AI 模式只是工作的一半。无菌、没有声音的写作和机器生成的内容一样明显。好的写作背后有一个真实的人。**

### 5.1 缺乏灵魂的写作迹象

- 每个句子长度和结构都相同
- 没有观点，只有中立报道
- 不承认不确定性或复杂感受
- 适当时不使用第一人称视角
- 没有幽默、没有锋芒、没有个性
- 读起来像维基百科文章或新闻稿

### 5.2 如何增加语调

| 技巧 | 说明 |
|------|------|
| **有观点** | 不要只是报告事实——对它们做出反应 |
| **变化节奏** | 短促有力的句子，然后是需要时间慢慢展开的长句 |
| **承认复杂性** | 「这令人印象深刻但也有点不安」胜过「这令人印象深刻」 |
| **适当使用"我"** | 「笔者认为…」或「让笔者困扰的是…」表明有真实的人在思考 |
| **允许一些混乱** | 完美的结构感觉像算法。跑题、题外话是人性体现 |
| **对感受要具体** | 不是「这令人担忧」，而是具体描述场景 |

### 5.3 示例对比

❌ **改写前(干净但无灵魂)**：
> 实验产生了有趣的结果。智能体生成了 300 万行代码。一些开发者印象深刻，另一些则持怀疑态度。影响尚不明确。

✅ **改写后(鲜活)**：
> 笔者真的不知道该怎么看待这件事。300 万行代码，在人类大概睡觉的时候生成的。开发社区有一半人兴奋不已，另一半人在解释为什么这不算数。真相可能在无聊的中间某处——但笔者一直在想那些通宵工作的智能体。

## 八、完整改写示例

### 8.1 示例 1：综合应用

❌ **原文(AI 特征明显)**：

> 此外，值得注意的是，大数据技术在精准营销中的应用具有重要的意义。首先，大数据技术能够实现消费者画像的精准构建。其次，大数据技术可以支撑营销决策的优化。最后，大数据技术有助于提升营销效果。综上所述，大数据技术是精准营销的核心支撑。

**问题分析**：
1. ❌ 「此外」「值得注意的是」— 模板化过渡
2. ❌ 「具有重要的意义」— AI 高频词汇
3. ❌ 「首先…其次…最后」— 模板化列举
4. ❌ 「综上所述」— AI 总结模板
5. ❌ 句长均匀(约 20 字)
6. ❌ 总分总结构

✅ **改写后(人性化 + 成语替换)**：

> 大数据技术对精准营销的价值已得到学界广泛认可。笔者在研究中发现，这种价值**举足轻重**，主要体现在三个层面。消费者画像的构建依赖数据。通过对用户行为数据的采集、整合与分析，企业得以勾勒出目标客户的完整面貌，画像精度**立竿见影**。营销决策的优化同样离不开数据支撑——基于数据分析的决策更加精准、高效。至于营销效果的提升，则是数据价值的直接体现，**成效卓著**。据张三(2023)的研究，应用大数据技术的企业营销 ROI 平均提升 2.3 倍。

**改写要点**：
1. ✅ 消除「此外」「值得注意的是」
2. ✅ 「具有重要」→「举足轻重」(成语替换)
3. ✅ 消除「首先…其次…最后」
4. ✅ 消除「综上所述」
5. ✅ 句长有明显波动(6字 → 45字)
6. ✅ 多种结构交替(陈述 → 定义 → 数据 → 引用)
7. ✅ 添加成语「立竿见影」「成效卓著」

## 九、质量自检清单

完成表达质量优化后，对照以下清单自查：

- [ ] 无「首先…其次…最后」式机械列点
- [ ] 已压缩「此外」「综上所述」「值得注意的是」等 AI 高频模板过渡词
- [ ] 已加入必要的自然承接语，如“具体来说、换句话说、放到实际使用里看”等，且承接后有实质内容
- [ ] 已删除机械冗余「的」字，但未把句子压缩到生硬难懂
- [ ] 已保留少量服务于自然度的轻冗余词，如“通常、往往、也会、在一定程度上”等，且未形成废话堆砌
- [ ] 句长有明显波动(标准差 > 10)
- [ ] 已形成长短句交织，而不是只把句子机械拉长或切短
- [ ] 信息密度高的句子已完成“抽主干 → 拆动作 → 补解释层”处理
- [ ] 段落开头形式多样
- [ ] 有适度的主观表达
- [ ] 有转折、让步等不完美感
- [ ] 标点使用多样(有分号、破折号)
- [ ] 适度使用成语替换(每段 2-3 个)
- [ ] 有近义词漂移与选词波动(每千字 3-5 处)
- [ ] 论述篇幅有不对称感
- [ ] 显性转接词已压缩，尽量不用“因此、同时、此外、从而”等可预测连接词
- [ ] 已使用句间语义承接、场景推进或对象切换替代模板化过渡
- [ ] 成语使用适度且贴合语境，未在技术细节附近堆砌
- [ ] 未触碰红线(无口语标记、无虚假引用等)
- [ ] 专业术语未被误替换
- [ ] 表达质量问题段落已完成局部修正
- [ ] 已按“场景化重写 → 结构重组 → 细节注入 → 自然承接与轻冗余 → 高密度句拆句解释 → 语言去模板化”的顺序处理，而不是先做机械同义替换
- [ ] 功能说明段已补入真实使用场景、系统响应路径或结果呈现方式
- [ ] 已祛除“背景意义—问题挑战—解决方案—价值总结”的八股文结构
- [ ] 若原文包含条款编号，改写后已保留相同编号、标题和顺序
- [ ] 若用户指出 100% 疑似条款，已优先进行单条款局部深改
- [ ] 已生成处理前计划和处理完成自检
- [ ] 已生成改写前后量化对比报告或说明无法生成的原因
- [ ] 已列出改写后自测清单，并标明“通过 / 需人工确认 / 未通过”
- [ ] 未编造原文没有提供的接口、数据表、实验指标或参考文献

## 十、注意事项

### 10.1 改写边界

- **保持语义不变**：改写不能改变原文核心观点
- **保护专业术语**：术语不在改写范围内
- **保持学术规范**：人性化不等于口语化
- **成语适度使用**：避免堆砌，每段不超过 2-3 个

### 10.2 风险提示

- 本地脚本结果仅用于辅助定位模板化表达、句式均匀和术语漂移问题
- 正式提交前仍需人工审校，确认语义、术语和引用准确
- 不同审校工具的规则不同，脚本结果不应作为唯一质量依据

### 10.3 伦理说明

- 本工具用于辅助写作，不是替代写作
- 用户应对最终内容负责
- 建议在改写基础上进行人工审核和修改

## 十一、AIGC 率降低标准流程

> AIGC 率降低的核心不是先替换词语，而是先改变段落生成逻辑。默认流程应先从真实使用场景切入，把清单式、模板式说明改成更符合本科论文写作习惯的自然段，再补入已有系统细节，最后才处理词语和句式层面的模板痕迹。

### 11.1 执行原则

所有 AIGC 降低任务均按本流程处理：

- 不把“同义替换”作为首要动作；同义替换只用于最后的局部修饰。
- 不把“首先、其次、最后”简单替换成其他连接词，而是重组段落表达方式。
- 压缩“因此、此外、综上”等可预测模板转接词，但允许使用“具体来说、换句话说、放到实际使用里看、对应到系统运行过程”等自然承接语完成解释和转场。
- 允许保留少量轻冗余词，例如“通常、往往、也会、在一定程度上”，用来打散过于规整的句式；但不得保留空泛套话或宣传式废话。
- 信息密度高的句子必须先拆主干，再拆动作，最后补解释层，避免只沿用原句骨架做词语替换。
- 允许在语义接近的前提下重构句子顺序和段内组织，不沿用原文句式骨架做机械替换。
- 成语可以适度使用，但只放在非技术性总结、边界说明或维护效果描述中；系统设计、实现、测试等技术细节附近不得堆砌成语。
- 不沿用“背景意义—问题挑战—解决方案—价值总结”的八股文套式；发现该结构时先改叙述组织。
- 功能说明类段落必须写清“用户为什么这样用、系统怎样响应、结果如何呈现”。
- 能补系统已有细节时优先补细节；信息不足时提示用户补充，不编造。

### 11.2 标准改写顺序

1. **先判定段落功能**：区分该段是在写用户需求、系统设计、功能实现还是测试结果。
2. **抽取真实场景**：补出使用者视角，例如“用户并不关心底层构建方式，而是希望快速找到资料”。
3. **识别并保护条款结构**：若原文包含（1）（2）（3）（4）等编号，必须保留编号、标题和顺序；不得为了降低模板感删除、合并或重排条款。
4. **条款内改写**：在每个条款内部改写为自然论文说明，减少“系统提供/系统支持/从而/确保”等模板链条。
5. **自然过渡与解释层**：压缩“因此、此外、从而”等高频模板词，但可加入“具体来说、放到实际使用里看、换句话说”等自然承接语，使段落转场更像人工解释。
6. **高密度句拆解**：遇到一句话塞入多个职责、动作或目标时，先抽主干，再拆动作，最后补一层解释说明。
7. **轻冗余控制**：允许少量“通常、往往、也会、在一定程度上”等缓冲词，但必须服务于承接和解释，不能变成废话。
8. **适度成语化处理**：可在非技术性表述中使用少量贴合语境的成语，如“有章可循”“防患于未然”等；不得在接口、参数、数据指标和测试结果附近堆砌成语。
9. **制造自然不对称**：重点功能可多写两句，次要功能一笔带过，避免每个功能点篇幅完全一致。
10. **复检模板词**：删掉“围绕该目标、功能支持、提升用户体验、具备一定”等泛化表达。
11. **保留本科论文语气**：表达可以自然，但不能口语化、不能写成宣传文案。

### 11.3 改写模板

**原始模板式写法：**

> 普通用户作为系统的主要使用群体，其核心目标在于高效获取知识信息与智能问答服务。系统需提供智能问答、语义检索、知识库访问和交互管理功能。

**深度人工化写法：**

> 普通用户是本系统最主要的服务对象。从实际使用场景来看，这类用户通常并不关心知识库底层的构建方式，而是希望能够快速找到自己需要的资料，并通过问答形式获得较为直接的解释。因此，系统在功能设计上需要围绕知识库访问、语义检索和智能问答展开，使用户在查询、浏览和连续追问过程中都能获得相对连贯的使用体验。

### 11.4 技术论文常用细节注入点

| 章节/段落 | 可注入细节 | 避免写法 |
|----------|------------|----------|
| 用户需求分析 | 用户操作目的、权限边界、查询路径、结果呈现方式 | 只写“用户可使用某功能” |
| 语义检索 | 向量化、相似度计算、排序、上下文片段 | 空泛写“提高检索效率” |
| 智能问答 | 先检索知识库再生成回答、引用来源、回答可信度 | 只写“大模型生成答案” |
| 知识库管理 | 知识库列表、分类、标签、关键词筛选 | 只写“支持管理知识库” |
| 多轮对话 | 连续追问、上下文范围、历史记录管理 | 只写“提升交互体验” |

### 11.5 输出要求

- 每次处理 AIGC 降低任务都必须输出三部分：**处理前计划、改写后文本、清单自检**。
- “处理前计划”需说明段落类型、主要风险和本轮处理策略；不得跳过计划直接改写。
- “改写后文本”优先给完整可替换版本，必要时再附简短改写要点。
- “清单自检”必须逐项标注“通过 / 需人工确认 / 未通过 / 不适用”。
- 自测清单至少包含：语义与术语未改变、条款编号与顺序保留、模板词清理、自然承接语使用、轻冗余控制、高密度句拆句解释、原句骨架重组、长短句变化、场景化与细节注入、未新增虚构信息、学术边界。
- 若任一关键项为“未通过”，应继续局部改写，不要直接交付最终版本。
- 不要承诺一定通过任何检测工具，只说明已降低模板化与机械表达特征。
- 若原文信息不足，应提示用户补充系统真实技术细节，而不是编造接口、数据库表或实验数据。

### 11.6 保留条款模式与 100% 疑似局部处理

当原文为条款式功能说明时，执行以下规则：

1. 保留原有编号、条款标题和功能项，不删除、不合并、不调整顺序。
2. 每个条款内部可以改变句式和叙述顺序，但核心含义必须保持一致。
3. 用户指出某一条款 100% 疑似时，只对该条款优先深改，不牵连其他已通过条款。
4. 高风险条款优先压缩解释性总结，减少“系统提供、系统支持、管理员可以、从而、确保”等模板词。
5. 可从真实运行场景切入，例如人员变化、权限边界、部署参数变化、资源占用、服务异常、日志查询等。
6. 不引入原文没有提供的接口、数据库表、告警机制、实验指标或外部工具。

#### 处理前计划模板

- 段落类型：需求分析 / 系统设计 / 系统实现 / 系统测试 / 摘要 / 其他。
- 原文结构识别：是否包含条款编号、条款数量、是否需要保留标题。
- 高风险点判断：模板化句式、重复主语、句长均匀、显性转接词、原句骨架残留。
- 本轮处理策略：场景化重写、结构重组、细节注入、语言去模板化的具体动作。
- 预期输出：改写文本、清单自检；有脚本输入时再补量化对比报告。

#### 处理完成自检模板

| 检查项 | 状态 | 说明 |
|---|---|---|
| 语义与术语未改变 | 通过 / 需人工确认 / 未通过 | 是否保留原意和关键术语 |
| 条款编号与顺序保留 | 通过 / 不适用 / 未通过 | 原文有编号时必须保留 |
| 模板词已压缩 | 通过 / 需人工确认 / 未通过 | 是否减少“确保、从而、围绕、功能支持”等表达 |
| 自然承接语使用 | 通过 / 需人工确认 / 未通过 | 是否加入“具体来说、换句话说、放到实际使用里看”等自然过渡，且未堆砌模板词 |
| 轻冗余控制 | 通过 / 需人工确认 / 未通过 | 是否保留少量“通常、往往、也会”等缓冲词，同时未制造废话 |
| 高密度句已拆解 | 通过 / 需人工确认 / 未通过 | 是否对职责、动作、目标过于集中的句子完成拆句和解释层补写 |
| 原句骨架已重组 | 通过 / 需人工确认 / 未通过 | 是否避免只做同义替换 |
| 长短句有变化 | 通过 / 需人工确认 / 未通过 | 是否避免每句长度接近 |
| 场景或系统细节充足 | 通过 / 需人工确认 / 未通过 | 是否用真实功能、权限、流程、日志、参数等支撑 |
| 未新增虚构信息 | 通过 / 需人工确认 / 未通过 | 是否未编造接口、表、指标、文献 |
| 学术语体稳定 | 通过 / 需人工确认 / 未通过 | 是否未口语化、宣传化或过度成语化 |
