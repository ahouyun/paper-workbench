# 论文初稿撰写工作流 (Paper Draft Workflow)

> 从用户的研究内容到完整初稿的系统化流程。

---

## 一、撰写前准备（必须完成）

### 1.1 用户需要提供的材料

**必须提供：**
- [ ] 研究问题描述（你要解决什么问题）
- [ ] 方法描述（你的核心创新是什么）
- [ ] 实验数据（数据集、基线、结果数字）
- [ ] 目标期刊（IEEE TITS/TKDE/TNNLS等）

**可选提供：**
- [ ] 已有图表
- [ ] 参考文献列表
- [ ] 导师/合作者的修改意见

### 1.2 材料收集清单

在开始撰写前，逐项确认：

| 材料 | 状态 | 说明 |
|------|------|------|
| 研究问题 | ☐ | 一句话描述 |
| 核心创新 | ☐ | 2-3点 |
| 数据集 | ☐ | 名称+规模 |
| 基线方法 | ☐ | 至少5个 |
| 实验结果 | ☐ | MAE/RMSE/MAPE |
| 消融实验 | ☐ | 至少3个组件 |
| 效率数据 | ☐ | 参数量/FLOPs/推理时间 |
| 目标期刊 | ☐ | 具体期刊名 |

---

## 二、撰写流程（按顺序执行）

### 步骤1：撰写Abstract（最后写，但先规划）

**输入：** 研究问题 + 核心创新 + 关键结果

**输出：** 150-200词的Abstract

**流程：**
1. 读取 `ieee-expression-patterns.md` 的Abstract首句模式
2. 根据用户研究选择合适的开头风格
3. 用"问题→方案→结果→影响"四段式结构
4. 检查是否避免了模板化表达
5. 检查是否有具体数字

**示例输出结构：**
```
[第1句] 任务定义+领域归属
[第2-3句] 现有方法的局限性
[第4-5句] 本文方案概述
[第6-7句] 关键实验结果
[第8句] 更广泛的影响
```

### 步骤2：撰写Introduction

**输入：** 研究问题 + 核心创新 + 相关工作

**输出：** 6段漏斗结构的Introduction

**流程：**
1. 读取 `chapter-patterns-ieee.md` 的六段式漏斗结构
2. 读取 `flexible-writing-guide.md` 的对话式写法
3. 按以下结构撰写：

```
第1段：领域重要性（1-2句，不废话）
第2段：技术挑战（具体化，非泛泛而谈）
第3段：现有方法分类与各自局限
第4段：现有方法的共同盲点（核心洞察）
第5段：本文方案概述与动机
第6段：贡献列表（具体、可操作）
```

**关键检查：**
- [ ] 是否避免了"With the rapid development of..."开头？
- [ ] 是否避免了"Despite significant progress"模板？
- [ ] 每个贡献是否绑定强动词+具体工件+证据位置？
- [ ] 是否使用了对话式写法而非模板化写法？

### 步骤3：撰写Related Work

**输入：** 相关论文列表

**输出：** 分类组织的Related Work

**流程：**
1. 将相关论文按技术路线分类
2. 每类写1-2段，指出共同局限
3. 最后一段说明本文与已有工作的区别

**关键检查：**
- [ ] 是否避免了简单的论文罗列？
- [ ] 是否指出了每类方法的共同局限？
- [ ] 是否说明了本文与已有工作的区别？

### 步骤4：撰写Method

**输入：** 方法描述 + 数学公式

**输出：** 完整的Method部分

**流程：**
1. 读取 `ieee-expression-patterns.md` 的Method写作模式
2. 按以下结构撰写：

```
4.1 Problem Formulation
    - 图定义：G = (V, E, A)
    - 输入输出张量：X ∈ R^(N×T×C)
    - 问题形式化：f_θ: X → X̂

4.2 Overall Architecture
    - 引用架构图
    - 列出主要组件

4.3 Module 1: [模块名]
    - 设计动机
    - 数学公式
    - 设计选择解释

4.4 Module 2: [模块名]
    - 同上

4.5 Training Objective
    - 损失函数定义
    - 训练策略
```

**关键检查：**
- [ ] 每个模块是否有设计动机？
- [ ] 数学公式是否完整且正确？
- [ ] 符号是否一致？
- [ ] 是否解释了设计选择的原因？

### 步骤5：撰写Experiments

**输入：** 实验数据 + 基线方法

**输出：** 完整的Experiments部分

**流程：**
1. 读取 `ieee-experiment-playbook.md` 的实验设计模式
2. 读取 `ieee-real-experimental-data.md` 的真实数据
3. 按以下结构撰写：

```
5.1 Experimental Setup
    - 数据集描述（传感器数、时间范围、采样率）
    - 基线方法（分类列举）
    - 实现细节（硬件、超参数、训练策略）

5.2 Main Results
    - 主结果表格
    - 结果分析（最好→次好→原因）

5.3 Ablation Study
    - 消融实验表格
    - 每个组件的贡献分析

5.4 Efficiency Analysis
    - 参数量、FLOPs、推理时间
    - 与基线的效率对比

5.5 Visualization (可选)
    - 预测曲线
    - 注意力热图
    - 案例分析
```

**关键检查：**
- [ ] 数据集描述是否包含传感器数、时间范围、采样率？
- [ ] 基线方法是否包含最新SOTA？
- [ ] 是否报告了15/30/60分钟三个时域？
- [ ] 消融实验是否覆盖所有核心组件？
- [ ] 是否报告了效率指标？

### 步骤6：撰写Conclusion

**输入：** 核心贡献 + 关键结果

**输出：** 简洁的Conclusion

**流程：**
1. 重述核心贡献（1-2句）
2. 总结关键实验结果（1-2句）
3. 承认局限性（1句）
4. 未来工作方向（1-2句）

**关键检查：**
- [ ] 是否避免了重复Abstract的内容？
- [ ] 是否承认了局限性？
- [ ] 未来工作是否与局限性呼应？

---

## 三、去模板化检查（每段完成后执行）

### 3.1 句子级别检查

- [ ] 是否每段开头都是"However,"或"In this paper,"？
- [ ] 是否每个贡献都是"We propose a novel..."？
- [ ] 是否每个结果都是"Extensive experiments demonstrate..."？
- [ ] 是否使用了具体的例子和数字？

### 3.2 段落级别检查

- [ ] 每段是否有自己的声音？
- [ ] 第一句话是否告诉读者这段要讲什么？
- [ ] 是否有过渡词连接？
- [ ] 最后一句是否与下一段衔接？

### 3.3 整体级别检查

- [ ] 是否有你自己的声音和风格？
- [ ] 是否承认了局限性？
- [ ] 是否有具体的例子和数字？

---

## 四、质量检查清单（提交前必须完成）

### 4.1 内容完整性

- [ ] Abstract是否包含问题、方案、结果、影响？
- [ ] Introduction是否包含6段漏斗结构？
- [ ] Method是否包含所有模块的描述？
- [ ] Experiments是否包含主结果、消融、效率分析？
- [ ] Conclusion是否包含贡献、结果、局限性、未来工作？

### 4.2 数据准确性

- [ ] 所有数字是否来自用户提供的实验数据？
- [ ] 所有引用是否通过验证？
- [ ] 表格中的数字是否与正文描述一致？

### 4.3 写作质量

- [ ] 是否避免了模板化表达？
- [ ] 是否使用了具体的例子和数字？
- [ ] 是否承认了局限性？
- [ ] 是否有你自己的声音和风格？

### 4.4 格式规范

- [ ] 是否符合IEEE格式要求？
- [ ] 图表是否清晰可读？
- [ ] 参考文献格式是否统一？

```

[Keywords]
1. Introduction
2. Related Work
3. Method
```

## 六、使用示例

**用户输入：**
```
研究问题：交通流预测中，现有方法忽略了交通拥堵的传播延迟
方法：提出传播延迟感知的Transformer
数据集：METR-LA, PEMS-BAY, PEMS04, PEMS08
基线：DCRNN, STGCN, GWNet, STAEformer
结果：METR-LA上MAE降低5.2%
目标期刊：IEEE TITS
```

**系统输出：**
```
1. 基于用户输入生成Abstract
2. 基于用户输入生成6段Introduction
3. 基于相关论文生成Related Work
4. 基于方法描述生成Method部分（含数学公式）
5. 基于实验数据生成Experiments部分（含表格）
6. 生成Conclusion
7. 输出质量报告
```
