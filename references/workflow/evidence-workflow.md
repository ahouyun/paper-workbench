# 证据链构建

当用户提供程序、仓库、数据库、API、截图或测试报告并需要写论文时使用。

## 目标

在写正文前，将实现证据转化为论文可用的事实。

## 优先输入

- 源码仓库或项目文件夹
- 学校模板或格式指南
- 任务书、开题报告、导师笔记
- 数据库 schema 或迁移文件
- API 文档或路由定义
- 截图和测试报告
- 部署或运行说明

## 证据构建

如果工作区刚初始化，运行：

```bash
python scripts/lifecycle.py --workspace thesis-workspace/ --build-evidence
```

输出：

```
paper-context/evidence/
  project-evidence.json
  code-structure.md
  tech-stack.md
  api-list.md
  database-schema.md
  test-results.md
```

脚本是初读。读完输出后，在做重要声明前检查源文件。

## 证据到论文映射

| 证据 | 论文用途 |
|------|----------|
| 目录树 | 系统组成、前后端分离、模块边界 |
| 包文件 | 技术栈和开发环境 |
| 路由/控制器 | API 设计和详细实现 |
| SQL/迁移/实体 | 数据库设计和 E-R 图 |
| 测试/报告 | 测试章节和结果可信度 |
| 截图 | 详细实现和测试图片 |

## 五层证据模型

不要直接跳到章节，使用五层：

1. **项目清单**：文件、技术标记、运行脚本、数据库/API 候选
2. **已验证事实**：确认功能或结果的源文件或用户材料
3. **论文事实**：标准化条目
4. **图表计划**：`figure-registry.yaml` 中的条目
5. **正文**：引用前几层的章节段落

## 系统论文起草顺序

1. 相关技术和开发环境
2. 需求分析
3. 总体设计
4. 数据库/API/模块设计
5. 详细实现
6. 测试
7. 引言和结论最后写

引言在知道真实贡献后更容易写，也更不假。

## 停止条件

如果以下项缺失，不写最终正文：

- 无学校模板或标准配置（布局敏感交付）
- 无代码证据（声称的模块）
- 无 schema/API 证据（数据库或接口声明）
- 无截图或运行证据（UI 声明）
- 无测试证据（"已测试"、"稳定"、"通过"、"有效"声明）

返回缺失材料清单。

## 图表规划

绘制前创建或更新 `figure-registry.yaml`：

- **系统架构**：基于真实目录和部署结构
- **模块图**：基于真实功能/模块边界
- **业务流程**：基于实际用户/管理员工作流
- **E-R 图**：基于 schema/实体
- **时序图**：基于控制器/服务/API 流程
- **截图**：基于实际运行页面或报告

不做无法追溯到代码或证据的装饰性图表。
