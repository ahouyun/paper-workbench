# 中文毕设论文写作规范

本文档合并了原 `writer_guidelines.md` 和 `writing_standards.md` 的核心内容，消除重复。

---

## 两阶段写作法（核心策略）

**绝不直接输出段落**，而是「先提纲 → 再成文」。此策略可有效：
- 降低 AI 模板化输出倾向
- 增加文本结构多样性
- 便于用户在要点阶段介入调整

### Stage 1：段落级要点规划

对每一章节，先生成结构化要点列表：

- 每段的核心论点（1 句话）
- 计划引用的文献（作者+年份）
- 支撑论据（数据/案例/理论）
- 段落间的逻辑衔接关系

**用户确认环节**：Stage 1 输出后，等待用户确认或调整，再进入 Stage 2。

### Stage 2：要点扩展为段落

1. **变换句式结构**：禁止连续 3 段使用相同的「总分」结构
2. **句长波动**：目标句长标准差 > 10
3. **自然引用**：将引用融入句子中，而非堆砌在段末
4. **段首多样性**：相邻段落首词不得相同

## 整体风格要求

### 学术正式语体

**推荐表达**：研究表明、据统计分析、笔者认为、从实证角度来看、基于上述分析、现有文献表明

**禁用 AI 特征词汇**：

| 禁用词汇 | 替代方案 |
|---------|---------|
| 此外 | 另外、同时、与之相伴 |
| 值得注意的是 | 需要关注的是 |
| 不可否认 | 必须承认、诚然 |
| 综上所述 | 总体而言、概括来说、总的来看 |
| 首先…其次…最后 | 直接陈述要点，或用「一方面…另一方面」 |
| 具有重要意义 | 意义重大、作用显著 |

### 论证结构多样性

避免每段都是「总分」结构，穿插使用：演绎法、归纳法、对比法、因果法、递进法。

### 括号规范

- 正文补充说明：英文括号 `()`
- 引用标注：方括号 `[]`
- **禁止使用中文括号**

---

## 段落写作规范

每个段落应包含：
- **主题句**：明确表达本段核心观点（20-40字）
- **展开句**：提供论据、数据、案例支持（100-200字）
- **小结句**：总结本段，呼应主题或承上启下（20-40字）

**段落长度**：150-300 字/段

---

## 引用规范

> **所有引用必须来自已验证文献池，禁止编造任何文献信息。**

### 引用硬约束

| 禁止行为 | 说明 |
|----------|------|
| 禁止编造文献标题 | 不能虚构论文标题 |
| 禁止编造作者姓名 | 不能使用占位符 |
| 禁止编造期刊名称 | 不能虚构期刊或会议名 |
| 禁止编造 DOI | 不能生成虚假 DOI 号 |
| 必须包含 DOI 链接 | 格式：`[DOI](https://doi.org/xxx)` |

### 引用频率

- 每千字至少 2 个文献引用（推荐 3-5 个）
- 各章节均有引用，不要集中在某一章
- **单句硬约束**：每句话最多引用 1 篇文献

### 引用格式（GB/T 7714）

严格遵循 GB/T 7714。快速参考：

- 期刊 `[J]`：`[序号] 作者. 题名[J]. 刊名, 年, 卷(期): 页码.`
- 学位论文 `[D]`：`[序号] 作者. 题名[D]. 保存地: 保存单位, 年份.`
- 专著 `[M]`：`[序号] 作者. 题名[M]. 出版地: 出版者, 出版年: 页码.`
- 会议 `[C]`：`[序号] 作者. 题名[C]//编者. 会议名, 地点, 时间: 页码.`
- 标准 `[S]`：`[序号] 标准编号, 标准名称[S].`

详细格式查阅 `prompts/reference_format_gbt7714.md`。

### 正文引用标注

- 标注位置：所引内容最末句的右上角
- **禁止同一句出现多个引用编号**（如 `[A,B]`、`[A-B]`）

## 代码展示规范

> 论文是"论述"而非"代码仓库"，代码展示必须精炼且有设计说明。

| 规则 | 标准 |
|------|------|
| 每段代码长度 | **≤ 20 行**，超出用 `// ...省略...` 代替 |
| 禁止完整类展示 | 不直接贴完整 Service、Controller 类 |
| 禁止 CRUD 展示 | 通用增删改查只描述不展示 |
| 关键代码优先 | 只展示核心算法、关键逻辑、创新实现 |

**代码展示三要素**：

1. **设计说明**（代码前）：为什么这样做，解决什么问题
2. **代码片段**（≤20行）：关键实现代码
3. **效果分析**（代码后）：达到了什么效果，有什么优势

## 图文排版铁律

### 排版顺序

```
文字描述 → 图表 → 代码（如适用）
```

禁止先放图表再写文字说明。

### 图名位置

- **图名在图的下方**，格式：`图X-X  名称`（短横线，编号与名称间隔两个半角空格）
- 示例：`图3-1  用户管理用例图`

### 表名位置

- **表名在表的上方**，格式：`表X.X  名称`（英文句点，编号与名称间隔两个半角空格）
- 示例：`表4.1  用户信息表结构`

### 图表说明

- 图表前必须有引导语：「如图X-X所示」「如表X.X所示」
- 图表后必须有 ≥50 字分析说明

## 章节写作要点

### 摘要（三段逻辑，550字左右）

- 第一部分：背景和本文目标（约150字）
- 第二部分：主要内容和技术路线（约150字）
- 第三部分：功能模块、成果和意义（约150字）
- 禁止出现：引用标记、图表编号、第一人称

### 引言（占总字数 10-15%）

研究背景 → 研究问题 → 研究意义 → 研究内容与方法

### 文献综述

- 按主题组织文献，不是简单罗列
- 指出现有研究的不足，引出本文贡献
- 近 5 年文献占比 > 50%

### 结论

- 概括主要研究发现（呼应研究问题）
- 理论贡献与实践启示
- 研究局限性（诚恳具体）
- 未来研究方向
- 不要提出新论点，不要简单重复摘要

## 七章制结构（软约束）

> 标准软件工程毕设7章结构，可根据学校要求调整。

```
第1章 绪论 → 第2章 关键技术 → 第3章 需求分析 →
第4章 系统设计 → 第5章 系统实现 → 第6章 系统测试 → 第7章 总结与展望
```

### 每章必备子节

| 章节 | 必备子节 |
|------|----------|
| 第1章 | 研究背景和意义、国内外研究现状、研究内容与方法、论文结构 |
| 第2章 | 后端技术、前端技术、数据库技术、开发环境 |
| 第3章 | 可行性分析（技术/经济/操作）、功能需求（含用例图）、非功能需求 |
| 第4章 | 系统架构设计、功能模块设计、数据库设计（含E-R图） |
| 第5章 | 各模块实现（含截图+代码） |
| 第6章 | 测试目的、功能测试（≥8条用例）、性能测试 |
| 第7章 | 总结、展望 |

---

## 篇幅分配建议（60-80页）

| 章节 | 目标页数 | 占比 |
|------|----------|------|
| 第1章 绪论 | 3-5 页 | 5-7% |
| 第2章 技术综述 | 4-6 页 | 6-8% |
| 第3章 需求分析 | 6-10 页 | 10-12% |
| **第4章 系统设计** | **12-18 页** | **20-25%** |
| **第5章 系统实现** | **10-15 页** | **15-20%** |
| 第6章 系统测试 | 5-8 页 | 8-10% |
| 第7章 总结展望 | 2-3 页 | 3-5% |

---

## 质量自检清单

- [ ] 每段都有明确的主题句
- [ ] 论证结构多样，不全是总分式
- [ ] 段落首句形式多样化
- [ ] 引用数量达标且分布均匀
- [ ] 无禁用的 AI 特征词汇
- [ ] 句长有明显波动
- [ ] 代码展示包含三要素
- [ ] 图表编号格式统一
