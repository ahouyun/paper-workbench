# 交通流预测审稿陷阱 (Traffic Prediction Review Pitfalls)

> TITS审稿人最常挑刺的问题，必须提前规避。

---

## 一、数据泄露 (Future Information Leakage)

### 1.1 什么是数据泄露

**定义：** 模型在训练时看到了本不应该看到的未来信息。

**常见形式：**
- 使用未来数据做归一化（用全局均值/方差，而非仅用训练集）
- 使用未来数据做特征工程（用未来天气预测当前交通）
- 数据划分不当（测试集信息泄露到训练集）

### 1.2 如何检测

**检查归一化：**
```python
# ❌ 错误：使用全局均值/方差
mean = all_data.mean()
std = all_data.std()

# ✅ 正确：仅使用训练集的均值/方差
mean = train_data.mean()
std = train_data.std()
```

**检查数据划分：**
```python
# ❌ 错误：随机划分
train, test = random_split(data, [0.8, 0.2])

# ✅ 正确：按时间顺序划分
train = data[:int(0.7*len(data))]
val = data[int(0.7*len(data)):int(0.8*len(data))]
test = data[int(0.8*len(data)):]
```

### 1.3 审稿人关注点

- 归一化是否仅使用训练集统计量？
- 数据划分是否按时间顺序？
- 特征工程是否使用了未来信息？

## 二、空间自相关的影响

### 2.1 问题

交通传感器之间存在空间自相关——相邻传感器的读数高度相似。这可能导致：
- 模型只是学习了"相邻传感器读数相似"，而非真正的时空模式
- 改进可能来自更好的空间平滑，而非更好的预测

### 2.2 如何应对

**检查方法：**
- 在不同空间粒度上评估（单个传感器 vs 区域平均）
- 报告在"困难"传感器上的性能（与其他传感器相关性低的）
- 使用空间交叉验证（留出整个区域而非随机传感器）

## 三、数据集划分的影响

### 3.1 不同划分方式的影响

| 划分方式 | 优点 | 缺点 |
|---------|------|------|
| 7:1:2 (METR-LA标准) | 与DCRNN一致 | 测试集可能不够长 |
| 6:2:2 (PEMS标准) | 更多验证数据 | 与部分论文不可比 |
| 按月划分 | 更真实的评估 | 不同月份交通模式不同 |

### 3.2 审稿人关注点

- 划分方式是否与标准一致？
- 是否报告了不同划分方式的结果？
- 测试集是否足够长以评估长期预测？

## 四、计算复杂度与实际部署

### 4.1 常见问题

**审稿人意见：** "The computational cost is too high for real-world deployment."

**应对方法：**
- 报告推理时间（ms/sample）
- 报告GPU内存占用
- 报告参数量
- 讨论是否可以部署在边缘设备

### 4.2 必须报告的效率指标

| 指标 | 说明 | 典型值 |
|------|------|--------|
| 参数量 (Params) | 模型参数数量 | 1-50M |
| FLOPs | 浮点运算次数 | 1-100G |
| 推理时间 | 单样本推理时间 | 1-100ms |
| GPU内存 | 训练时GPU占用 | 1-16GB |
| 训练时间 | 总训练时间 | 1-24h |

## 五、实验设置的可比性

### 5.1 问题

不同论文使用不同的实验设置，导致结果不可比：
- 不同的数据预处理
- 不同的超参数搜索范围
- 不同的随机种子
- 不同的评估指标

### 5.2 如何确保可比性

**必须报告：**
- 数据预处理方法（归一化、缺失值处理）
- 超参数搜索范围
- 随机种子
- 评估指标的计算方式

**使用统一框架：**
- LibCity: https://github.com/LibCity/Bigscity-LibCity
- BasicTS: https://github.com/GestaltCogTeam/BasicTS

## 六、消融实验的常见陷阱

### 6.1 不公平的消融

**错误：**
```
Full Model: 8.2M params, MAE 2.49
w/o Module A: 5.1M params, MAE 2.58
```
参数量不同，无法判断改进来自模块A还是参数量增加。

**正确：**
```
Full Model: 8.2M params, MAE 2.49
w/o Module A: 8.2M params, MAE 2.58 (用相同大小的FFN替代)
```

### 6.2 消融不充分

**错误：** 只消融2-3个组件

**正确：** 消融所有核心组件，包括：
- 空间模块
- 时间模块
- 注意力机制
- 归一化方式
- 损失函数

## 七、常见审稿意见及应对

### 7.1 "改进不显著"

**审稿人意见：** "The improvement over STAEformer is only 0.05 MAE, which is marginal."

**应对：**
- 提供统计显著性检验（p < 0.05）
- 计算效应量（Cohen's d）
- 在不同数据集上验证一致性
- 讨论改进的实际意义

### 7.2 "实验设置不公平"

**审稿人意见：** "The comparison is unfair because different methods use different settings."

**应对：**
- 使用统一框架（LibCity/BasicTS）
- 报告所有方法的超参数
- 使用相同的随机种子

### 7.3 "缺乏可复现性"

**审稿人意见：** "The results cannot be reproduced."

**应对：**
- 公开代码
- 报告所有超参数
- 报告随机种子
- 提供详细的运行说明
