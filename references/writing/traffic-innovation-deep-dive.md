## 一、因果推断：从相关性到因果性（30+篇论文）

### 1.1 核心问题

大多数交通预测方法学习的是**相关性而非因果性**。现有图结构通常基于距离或连通性预定义，无法区分"上游拥堵导致下游拥堵"（因果）和"两条路同时拥堵因为都在同一区域"（混杂）。

### 1.2 因果图发现

|------|------|---------|--------|
| Molavipour et al. | 2020 | Directed Information Graph | 非线性非高斯因果图，比Granger更通用 |
| STGC-GNNs (Physica A) | 2023 | Spatial-Temporal Granger Causality | 全局动态信息替代局部静态依赖 |
| DCGCN (IEEE CASE) | 2023 | Time-varying Dynamic Bayesian Network | 动态因果图随时间演化 |
| CAL (arXiv) | 2024 | Causal Adjacency Learning | OOD泛化的因果邻接矩阵 |
| SPACY (arXiv) | 2024 | Variational Inference for Latent Causal Graph | 隐空间因果发现，假设更少 |
| CGProNet (arXiv) | 2024 | Causal Graph Processes + GNN | 高阶图滤波器，因果可解释 |
| CausalNet (arXiv) | 2024 | Granger Causality + Self-Correction | 动态因果图+自纠正模块 |
| **ST-HCMs** | **2025** | **AAAI** | **时空层次因果模型** | **首个形式化ST因果扩展，Collapse定理** |

- Collapse定理：复杂ST-HCM随数据增加收敛为简单扁平因果模型
- 支持未观测的时间不变单元级混杂下的因果识别

### 1.7 因果推断方向的创新机会

1. **因果图替代预定义邻接矩阵**: 用Granger因果或PC算法从数据中学习有向空间依赖
2. **反事实预测**: "如果这条路封闭/事故/施工，交通会怎样？"——几乎空白
3. **不变因果预测**: 找到跨环境不变的因果特征，提升OOD泛化
4. **LLM+因果**: Event-CausNet开创性地用LLM从文本提取因果知识
5. **TraffiDent数据集**: 首个专用因果交通分析基准

**Dynamic Aware关键发现**：
- 分布内样本的预测误差呈现模态依赖分布，且随时间演化
- OOD检测应考虑交通预测误差的多模态特性

## 五、物理信息深度学习（35篇论文）

### 5.1 核心问题

纯数据驱动方法可能产生物理不合理的预测（如流量突然消失或凭空产生）。物理信息深度学习将交通流物理模型（LWR、CTM、基本图关系）与神经网络结合，在保持预测精度的同时保证物理合理性。

### 5.2 物理约束损失函数

| 论文 | 年份 | 核心方法 | 创新点 |
|------|------|---------|--------|
| CLINN (arXiv) | 2023 | 守恒律约束神经网络 | LWR守恒律作为软约束，稀疏传感器下效果显著 |
| PINN-Traffic (PRR) | 2023 | PINN框架+交通流PDE | 首次系统性验证PINN在交通状态估计中的有效性 |
| Non-local LWR (arXiv) | 2024 | 非局部LWR模型+神经网络 | 非局部守恒律处理驾驶员预期行为 |
| SPIDL (arXiv) | 2024 | 随机物理信息深度学习 | Ito型随机LWR模型处理交通不确定性 |
| Physics-Constrained STG (IEEE TITS) | 2024 | 时空图+物理约束 | 图网络中的物理一致性损失 |
| PI-GCN (arXiv) | 2023 | 物理信息图卷积 | 基本图关系作为图卷积归纳偏置 |
| LWR-Net (TRC) | 2023 | 可学习LWR参数 | 数据驱动估计LWR模型参数（自由流速度、阻塞密度） |
| Conservation-Law Loss (IEEE Access) | 2023 | 守恒律正则化 | 将∂ρ/∂t + ∂(ρv)/∂x = 0作为正则项 |
| FDM-Informed (arXiv) | 2024 | 基本图约束 | 流量-密度-速度三角关系作为软约束 |

**创新点**: 将LWR守恒律 ∂ρ/∂t + ∂(ρ·v(ρ))/∂x = 0 作为损失函数的正则项，确保车辆数量守恒。

**关键发现**: 物理约束在稀疏传感器数据场景下效果最为显著——当传感器间距较大时，纯数据驱动方法容易产生不合理的插值，而物理约束可以引导模型给出物理合理的预测。

### 8.5 LLM的独特优势

1. **零样本/少样本预测**：LLMTime证明零样本可匹配专用模型，UrbanGPT在数据稀缺场景表现优异
2. **跨域迁移**：UniST单一模型处理交通流/人流/犯罪等多种城市任务，TransLLM跨7个数据集验证
3. **多模态融合**：CrossTrafficLLM融合文本语义与路网结构，Event-CausNet从文本提取因果知识
4. **可解释性**：CrossTrafficLLM同时预测和生成自然语言解释，CoT-LLM通过链式推理提供预测依据
5. **统一框架**：一个模型解决多种城市时空预测任务，降低部署成本

### 8.8 LLM方向的创新机会

## 九、Mamba/SSM用于交通预测（17篇论文）

### 9.1 核心问题

Transformer的O(L²)复杂度在长序列交通预测中计算成本高昂。Mamba（Gu & Dao, 2023）通过输入依赖的选择机制和线性时间复杂度O(L)，在长序列建模方面展现出显著优势。2024年起，大量研究将Mamba引入时空交通预测，形成了GNN（空间）+ Mamba（时间）的主流范式。

### 9.3 Mamba vs Transformer 对比

| 对比维度 | Transformer | Mamba/SSM | 结论 |
|---------|------------|-----------|------|
| 时间复杂度 | O(L²) | O(L) | Mamba在长序列上显著优势 |
| 空间复杂度 | O(L²) | O(L) | Mamba内存效率更高 |
| 参数量 | 较多 | 少30-60% | Mamba更轻量 |
| 短期预测(15-30min) | 有竞争力 | 接近 | 差异不大 |
| 中期预测(30-60min) | 中等 | 优势显现 | Mamba开始领先 |
| 长期预测(60min+) | 性能下降 | 优势显著 | Mamba核心优势 |
| 可解释性 | 注意力图直观 | 选择性扫描不直观 | Transformer更易解释 |
| 大规模路网(10000+节点) | 有验证 | 验证不足 | Transformer更成熟 |

### 9.4 空间扫描策略对比

| 策略 | 代表论文 | 方法 | 优劣 |
|------|---------|------|------|
| 无图建模 | ST-Mamba | ST-Mixer直接处理时空 | 简单但可能丢失拓扑信息 |
| 随机游走 | SpoT-Mamba | 节点特定随机游走序列 | 保留局部结构，但随机性 |
| 图卷积融合 | MGCN, GAMMA-Net | GCN/GAT处理空间+Mamba处理时间 | 主流范式，效果稳定 |
| DTW聚类 | DKGCM | DTW+K-means聚类后图卷积 | 数据驱动的空间分组 |
| 图傅里叶 | STGFMamba | 图傅里叶变换+Mamba | 频域分析增强 |
| 领域自适应 | Damba-ST | 共享+独立领域子空间 | 支持跨域迁移 |

### 9.5 Mamba方向的创新机会

1. **Mamba+扩散模型**：Mamba的高效序列建模+扩散的概率生成，几乎空白
2. **Mamba+因果推断**：E2-CSTP开创，选择性扫描天然过滤因果无关信息
3. **大规模路网验证**：10000+节点的城市级验证几乎缺失
4. **在线实时预测**：Mamba的O(L)复杂度适合实时场景，但验证不足
5. **多模态融合**：天气、事件等外部因素的Mamba原生融合方式待探索
6. **物理信息Mamba**：将交通流物理约束嵌入选择性扫描机制
7. **统一benchmark**：缺乏标准化评估，难以公平比较不同Mamba变体

### H.1 跨城市模型合并 / 拆分（任务算术与权重解耦视角）

**适用场景**：老师或合作者给出的方向不是“再提一个更深的单城模型”，而是“多个城市已经各自有模型，现在要区域一体化、跨城协同、隐私受限共享、边缘落地”。这类题目更适合讲“模型如何合并与再拆分”的故事，而不是继续讲单点预测精度。

**一句话问题定义**：
- 已有城市模型 `M_A`, `M_B`, `M_C` 分别在本地训练完成；
- 现在需要一个区域级模型 `M_union`；
- 目标是在**不从 scratch 联合重训**或只付出极小增量训练代价的前提下，实现跨城市知识合并；
- 随后再把 `M_union` 拆成城市/区域/场景专属版本。

**推荐叙事模板**：
1. 原有交通系统是分城市、分部门、分路网维护的，模型天然碎片化。
2. 新需求来自大湾区交通一体化、跨城事件联动、统一调度平台或隐私受限的数据协同。
3. 困难不在“是否有更多数据”，而在“已有模型参数、表示和知识如何可复用地合并”。
4. 如果能合并，就自然存在反向问题：如何在统一模型形成后，再拆回个性化、本地化、低时延的小模型。

**技术路线拆解**：
- 权重空间合并：
  - task arithmetic
  - weight decoupling
  - adapter / LoRA fusion
  - Fisher 或 importance-aware merge
- 表示空间合并：
  - latent alignment
  - graph optimal transport
  - parameter matching / neuron matching
  - shared encoder + city-specific head
- 受限共享场景：
  - federated averaging
  - personalized FL
  - federated distillation
  - federated foundation model
- 拆分路线：
  - distill `M_union -> M_A' / M_B' / M_C'`
  - 按区域聚类、社区检测、功能区分层拆分
  - 用 MoE/router/prompt 在一个统一模型内部做“软拆分”

**与现有交通方向的连接点**：
- 和 `UniST / OpenCity / MoST / UrbanFM` 这类基础模型工作相连时，故事偏“统一骨干 + 城市适配”。
- 和 `FedTransfer Cross-City / SecureGraphFL / FGNNEH / Federated GFM` 相连时，故事偏“隐私约束下的模型合并”。
- 和 `跨城市元学习 / MGSTA / FML-TFP / Meta-Learning MFD` 相连时，故事偏“学一个可快速合并/快速适配的初始化”。
- 和 `Cross-City Latent Alignment / Graph OT` 相连时，故事偏“先对齐再合并”。

**实验设计最小骨架**：
1. 先选有公开代码、能稳定复现的 backbone，不要一开始就做超大基础模型。
2. 至少准备 `A/B/C` 三个城市或三个区域级数据划分。
3. 比较：
   - pooled-data scratch training
   - single-city training
   - fine-tune from one city to another
   - weight merge / adapter merge
   - federated merge
   - meta-init + few-shot adapt
4. 除 `MAE/RMSE/MAPE` 外，增加：
   - merge cost（额外训练轮次、通信成本、参数增量）
   - forgetting（合并后原城市性能回退）
   - adaptation speed（拆分后收敛速度）
   - personalization gap（统一模型到本地模型的性能差）

**适合写成创新点的条件**：
- 真正展示了“不从 scratch 合并”的可行性，而不是简单把数据并起来重训。
- 合并和拆分至少有一边给出明确机制，另一边给出有说服力的实验或案例。
- 能说明为什么该问题在交通场景下比通用多任务学习更难，例如路网异构、传感器语义不一致、行政边界和隐私约束。

**不要误写**：
- “多城市联合训练”不等于“模型合并”。
- “从区域大模型微调到单城市”不自动等于“模型拆分”，除非明确给出拆分机制或资源收益。
- 如果只有概念图，没有真实参数合并、对齐、蒸馏、联邦或元学习协议，就只能写成 future work。


### H.2 Machine Unlearning / 污染参数修复（从持续学习到可遗忘学习）

**适用场景：** 前面 H.1 已经讲了“多个城市已有模型如何合并，再如何拆分”。接下来很自然的反向问题是：如果某些节点、区域或客户端后来被证明是坏的、漂移的、被投毒的、需要撤回的，那么统一模型里已经吸收进去的那部分知识，如何**不重训全模型**地删掉。

**一句话问题定义：**
- 某些传感器节点损坏，持续输出错误值；
- 某些区域数据被攻击或投毒；
- 某个城市后来不允许继续共享历史数据；
- 某一时间段的数据已经明显过时；
- 问题不再是 anomaly detection，而是模型已经被污染以后，怎样做 **selective forgetting / model repair**。

**为什么这个方向好讲故事：**
1. 持续学习解决“怎么把新知识加进去”。
2. 机器遗忘解决“怎么把错误知识拿出来”。
3. 合并解决“知识如何汇聚”。
4. 拆分解决“知识如何重新个性化”。
5. 四者连在一起，就不是一个小修小补模块，而是完整的区域级交通模型生命周期。

**可直接挂靠的近期参考：**
- `Machine Unlearning of Traffic State Estimation and Prediction` ([arXiv:2507.17984](https://arxiv.org/abs/2507.17984))：可以直接支撑“privacy / poisoned / outdated data 都可能触发 unlearning”这条论证线。
- `Continuous Learning for Android Malware Detection` (USENIX Security 2023)：可借它的系统叙事，不是照搬任务，而是借“概念漂移、持续更新、自污染风险、修复闭环”的写法。

**交通场景下可以怎么具体化：**
- 交通流预测：坏传感器长期漂移，把局部速度/流量模式带偏。
- 交通状态估计：局部观测异常导致全局状态重建被污染。
- OD / 出行需求：节假日或突发事件后的过时数据残留在模型里。
- 轨迹/移动流量：某些区域的隐私数据需要删除，必须从模型中遗忘。

**最小技术路线：**
- 先选一个有公开代码、容易跑通的 backbone，如 `STGCN / DCRNN / Graph WaveNet / STAEformer`。
- 再把“大框架”作为一号创新，把“在该 backbone 上实现局部 unlearning 或 repair”作为二号创新。
- 不要一上来就追求自建超大模型，否则实验和故事都会变散。

**可写的 unlearning 机制：**
- 节点级 / 区域级定位后做 localized adapter reset
- 近似 influence rollback 或梯度反向补偿
- city-specific head 替换 + shared encoder 保留
- teacher-student re-distillation，用净化数据重新蒸馏
- 对统一模型先遗忘，再蒸馏拆分到本地模型

**实验设计建议：**
1. 选 `A/B/C` 三个城市或三个区域子图。
2. 人为构造污染：
   - 节点故障 / 传感器漂移
   - 标签翻转 / 数据投毒
   - 局部时间段失真
   - 隐私删除请求
3. 比较：
   - full retrain
   - naive fine-tune
   - localized unlearning
   - parameter-efficient unlearning
   - merge 后再 unlearn
4. 除 `MAE/RMSE/MAPE` 外增加：
   - forgetting effectiveness
   - utility retention
   - repair cost
   - affected-region recovery
   - unaffected-region collateral damage

**特别适合写成亮点的地方：**
- 不是泛泛讲“模型鲁棒性”，而是讲“模型已经被污染后如何修复”。
- 不是只讲单城预测，而是讲“跨城市合并后的统一模型如何定向遗忘某一部分知识”。
- 不是只适用于交通流，而是可以平移到交通状态估计、OD、轨迹、移动流量，甚至其他时空学习任务。

**推荐记忆模板：**
- “合并是把知识收进来，持续学习是把新知识加进去，unlearning 是把坏知识拿出去，拆分是把统一知识重新分发出去。”
